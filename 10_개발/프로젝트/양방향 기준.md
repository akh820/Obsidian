## 양방향 기준 (간략)

**테이블 구조가 아니라 "사용 패턴"으로 결정!**

### 현재 관계

```
EquipmentRequest ──1:N── RequestItem ──N:1── Equipment
     (양방향)                               (단방향)
```

---

### 왜 이렇게?

**EquipmentRequest ↔ RequestItem (양방향)**

```java
// 자주 쓰는 패턴
request.getItems();  // 요청의 아이템 목록 필요 ✅
// + cascade로 함께 저장/삭제
```

**Equipment → RequestItem (단방향)**

```java
// 거의 안 쓰는 패턴
equipment.getRequestItems();  // 이 비품이 포함된 모든 요청? 잘 안 씀 ❌

// 필요하면 Repository로 조회
requestItemRepository.findByEquipment(equipment);  // 이렇게 하면 됨
```

---

### 정리

|관계|방향|이유|
|---|---|---|
|Request ↔ Items|양방향|함께 조회/저장/삭제 자주 함|
|Equipment ← Item|단방향|Item에서 Equipment만 조회하면 충분|

**핵심: "이 방향으로 자주 조회하나?" 기준으로 결정!**

양방향은 필요할 때만! 무조건 걸면 복잡해짐.
